# 동료평가 대비 노트 — 시스템 관제 자동화 (미션 13)

> 이 문서는 **내용을 모르는 상태에서 평가자 앞에서 방어**하기 위한 학습 노트다.
> 각 구성요소마다 "왜 필요한가 + 어떻게 동작하는가"를 30초 안에 말할 수 있어야 한다.

## 0. 한 줄 요약 (오프닝용)

> "리눅스 서버를 운영자 관점에서 직접 설계했습니다. **SSH 보안 강화 → 방화벽 → 역할별 계정/권한 → 환경변수 표준화 → 쉘 스크립트 자동 관제 → cron 매분 실행**까지, 서버가 죽거나 자원이 튀는 걸 자동으로 잡아내는 파이프라인을 구축했습니다."

이 한 줄을 발표 첫 문장으로 던지면 그 다음은 평가자가 항목별로 묻는 흐름이 된다.

## 1. SSH — 포트 변경 + Root 차단

### 왜 필요한가
- 기본 포트 **22번**은 인터넷 봇이 24시간 무차별 대입 공격(brute force)을 시도하는 1순위 표적이다.
- 포트를 **20022**처럼 비표준으로 옮기면 자동화된 스캔의 90% 이상이 그냥 지나친다. (= **noise reduction**)
- Root 원격 로그인을 차단하면, 공격자가 비밀번호 한 번에 시스템 전체 권한을 얻는 경로를 끊는다. 일반 계정 → `sudo`로 한 단계 거치게 강제한다.

### 어떻게 동작하는가
- `/etc/ssh/sshd_config` 파일의 두 줄:
  - `Port 20022`
  - `PermitRootLogin no`
- 수정 후 `sudo systemctl restart ssh`로 적용.
- 검증: `ss -tulnp | grep 20022` 로 sshd가 새 포트에서 LISTEN하는지 확인.

### 자주 듣는 질문
- Q. "포트만 옮기면 보안이 되나? **Security by obscurity** 아닌가?"
  - A. "맞습니다. **그 자체로는 보안이 아니라 노이즈 감소**입니다. 하지만 로그 양이 줄면 진짜 공격을 더 빨리 탐지할 수 있고, 방화벽·키 인증과 **layered defense**의 한 층으로 의미가 있습니다."
- Q. "왜 하필 20022?"
  - A. "관례적으로 22 끝자리를 유지해 SSH임을 운영자가 기억하기 쉬운 비표준 포트로 정한 것입니다. 1024 이상이라 권한도 단순합니다."

## 2. 방화벽 (UFW) — 20022 / 15034 만 허용

### 왜 필요한가
- **Default Deny** 원칙 — 명시 허용한 것 외에는 전부 차단. 공격면을 최소화.
- 앱 포트(15034)와 SSH 포트(20022) 두 개만 열면, 외부에서 닿을 수 있는 공격 경로가 정확히 두 개로 한정된다.

### 어떻게 동작하는가
```bash
sudo ufw default deny incoming      # 인바운드 전부 차단
sudo ufw default allow outgoing     # 아웃바운드는 허용
sudo ufw allow 20022/tcp            # SSH만
sudo ufw allow 15034/tcp            # 앱만
sudo ufw enable
```
- 검증: `sudo ufw status verbose` 에서 두 포트만 ALLOW로 보여야 한다.

### 자주 듣는 질문
- Q. "왜 firewalld가 아니라 UFW?"
  - A. "Ubuntu 표준이고 문법이 단순합니다. 본 환경(Ubuntu 22.04)에 가장 적합합니다. RHEL 계열이면 firewalld였을 겁니다."
- Q. "아웃바운드는 왜 다 열어두나?"
  - A. "이 서버가 외부 API 호출, 패키지 업데이트 등 정상 작업을 못 막으면 운영 불가입니다. 인바운드만 제한해도 외부 침입 방어 목적은 달성됩니다."

## 3. 계정 / 그룹 / ACL — 역할 분리

### 구조
| 계정 | 역할 | 소속 그룹 |
|---|---|---|
| `agent-admin` | 운영자 | agent-common, **agent-core** |
| `agent-dev` | 개발자 | agent-common, **agent-core** |
| `agent-test` | 테스터 | agent-common |

| 그룹 | 접근 가능 자원 |
|---|---|
| **agent-common** (3명 전원) | `upload_files/` — 모두 R/W |
| **agent-core** (admin + dev만) | `api_keys/`, `/var/log/agent-app/` — 민감자원 |

### 왜 필요한가
- **Principle of Least Privilege (최소 권한 원칙)** — 테스터는 API 키나 로그를 볼 이유가 없다. 사고 시 영향 범위를 줄인다.
- 권한을 **개인이 아니라 그룹**으로 묶어두면, 인사이동이 있을 때 그룹 멤버십만 바꾸면 된다. (확장성)

### 어떻게 동작하는가
- `useradd`로 계정 3개, `groupadd`로 그룹 2개 생성.
- `usermod -aG <그룹> <계정>` 으로 멤버십 부여.
- 디렉토리에 `chgrp agent-core api_keys && chmod 770 api_keys` → agent-core 그룹원만 읽기/쓰기.
- 검증: `id agent-test` 했을 때 `agent-core`가 **안 보여야 한다**.

### 자주 듣는 질문
- Q. "ACL과 그룹 권한의 차이는?"
  - A. "그룹 권한은 한 파일에 **한 그룹**만 지정 가능. ACL(`setfacl`)은 여러 그룹·여러 사용자에게 세밀한 권한을 동시에 줄 수 있습니다. 이 미션은 그룹 두 개로 충분히 표현되어서 ACL은 쓰지 않았습니다."
- Q. "왜 admin과 dev에 같은 권한을 줬나?"
  - A. "운영자와 개발자 모두 장애 대응 시 로그와 키에 접근해야 하기 때문입니다. 테스터는 결과만 보면 되므로 분리했습니다."

## 4. 디렉토리 권한 — `770`의 의미

### 구조
```
/home/agent-admin/agent-app/
├── upload_files/   agent-admin:agent-common  770   (전원 R/W)
├── api_keys/       agent-admin:agent-core    770   (admin+dev only)
│   └── t_secret.key  agent-admin:agent-core  640
└── bin/
    └── monitor.sh    agent-dev:agent-core    750   (admin+dev 실행)
/var/log/agent-app/  agent-admin:agent-core   770   (admin+dev R/W)
```

### 권한 숫자 읽는 법
- `7` = rwx (읽기+쓰기+실행)
- `5` = r-x (읽기+실행)
- `0` = --- (없음)
- **세 자리** = owner / group / others
- `770` → "소유자/그룹 모든 권한, 외부인 0" → 그룹 멤버십이 출입증
- `640` → "owner는 R/W, group은 R, others 0" → 키 파일은 **쓸 일이 없으니 R만**
- `750` → "owner R/W/X, group R/X, others 0" → 스크립트는 **실행 가능**해야 하니 X 포함

### 자주 듣는 질문
- Q. "monitor.sh를 왜 owner를 agent-dev로 했나?"
  - A. "**SoD (Separation of Duties)** — 개발자가 짠 스크립트를 운영자가 실행한다는 구조입니다. agent-dev가 파일을 짜고, agent-admin이 agent-core 그룹원으로서 실행하는 흐름이 자연스럽습니다."

## 5. 환경변수 — 실행 환경 고정

### 변수 6개
```bash
AGENT_HOME=/home/agent-admin/agent-app
AGENT_PORT=15034
AGENT_UPLOAD_DIR=$AGENT_HOME/upload_files
AGENT_KEY_PATH=$AGENT_HOME/api_keys/t_secret.key
AGENT_LOG_DIR=/var/log/agent-app
```
- 키 파일 내용: `agent_api_key_test`

### 왜 필요한가
- 경로/포트를 **코드에 하드코딩하지 않고** 외부 변수로 빼면, 같은 코드를 dev/stage/prod 어디서나 재사용 가능. (**12-Factor App**)
- 새 셸이 열릴 때마다 자동 로드되어야 하므로 `/etc/profile.d/agent-app.sh` 에 두었다.

### 어떻게 동작하는가
- `/etc/profile.d/*.sh` 는 로그인 셸이 시작할 때 자동 source됨.
- **cron은 예외!** cron은 비대화 셸이라 profile.d를 안 읽는다. 그래서 crontab 라인 앞에 환경변수를 명시 주입했다.
  ```
  * * * * * AGENT_HOME=... AGENT_PORT=15034 ... /home/agent-admin/agent-app/bin/monitor.sh
  ```

### 자주 듣는 질문
- Q. "키를 환경변수가 아닌 파일로 둔 이유?"
  - A. "환경변수는 `ps -eo environ` 같은 명령으로 다른 프로세스가 노출시킬 수 있습니다. 파일은 권한 640으로 보호하면 더 안전합니다."

## 6. 앱 부트 시퀀스 5단계

### 단계
```
[1/5] Checking User Account               [OK]   ← root 아닌 agent-* 계정인지
[2/5] Verifying Environment Variables     [OK]   ← 6개 변수 모두 있는지
[3/5] Checking Required Files             [OK]   ← t_secret.key 존재?
[4/5] Checking Port Availability          [OK]   ← 15034 바인드 가능?
[5/5] Verifying Log Permission            [OK]   ← /var/log/agent-app 쓰기 가능?
Agent READY
```

### 왜 5단계인가
- **앱이 죽기 전에 미리 실패해야 한다** (fail-fast). 운영 중 갑자기 "키 파일 없음"으로 죽으면 추적이 어렵다. 시작 시점에 한 번에 체크.
- 각 단계가 한 가지만 검사해서, FAIL이 나오면 **무엇이 문제인지 즉시 보인다**.

### 자주 듣는 질문
- Q. "Boot Sequence가 통과해도 운영 중 환경이 바뀌면?"
  - A. "그래서 monitor.sh가 매분 health check를 합니다. 시작 시 검증 + 운영 중 모니터링, 두 단계 모두 필요합니다."

## 7. monitor.sh — 매분 자동 관제

### 흐름 (위에서 아래로)
1. **HEALTH CHECK**
   - `pgrep -f agent_app.py` → 프로세스 살아있는지
   - `ss -tln | grep :15034` → 포트 LISTEN 중인지
   - 둘 중 하나라도 실패 → **exit 1** (cron이 cron.log에 실패 기록)
2. **FIREWALL CHECK** (경고만)
   - `ufw status` 비활성이면 `[WARNING]` 출력하고 **계속 진행** (관제 자체는 막지 않음)
3. **RESOURCE 수집**
   - CPU/MEM: `ps -p PID -o %cpu,%mem`
   - DISK: `df / | awk 'NR==2{print $5}'`
4. **임계값 경고**
   - CPU > 20%, MEM > 10%, DISK > 80% 중 어느 하나 초과 시 `[WARNING]` 출력
5. **로그 기록**
   - `/var/log/agent-app/monitor.log` 에 한 줄 append:
     `[YYYY-MM-DD HH:MM:SS] PID:.. CPU:..% MEM:..% DISK_USED:..%`
6. **로그 회전**
   - 10MB 도달 시 `.1`, `.2`, ... 로 밀어내고 11번째는 삭제. **최대 10개 보관**.

### 권한
- 파일: `750`, owner=**agent-dev**, group=**agent-core**
- 실행 주체: cron이 **agent-admin**으로 매분 호출 (agent-admin은 agent-core 그룹원이라 실행 가능)

### 자주 듣는 질문
- Q. "왜 부동소수 비교를 awk로 했나?"
  - A. "bash는 정수 비교만 됩니다 (`[ $cpu -gt 20 ]`). CPU가 `15.3%`처럼 소수라 `awk 'BEGIN{exit !(c>t)}'` 로 처리했습니다."
- Q. "방화벽이 꺼져있어도 왜 exit 안 하나?"
  - A. "방화벽은 보안이지 앱 동작과 무관합니다. 앱이 죽은 게 아니므로 관제 자체는 계속되어야 합니다. [WARNING]으로 알리는 게 맞습니다."
- Q. "로그 회전을 왜 logrotate 안 쓰고 스크립트 안에 넣었나?"
  - A. "**제약 조건 — Bash로만 작성**. logrotate도 가능하지만 외부 의존을 줄이고 스크립트 단독으로 완결되게 했습니다. 둘 다 정답입니다."

## 8. cron — 매분 실행

### 라인
```
* * * * * AGENT_HOME=... AGENT_PORT=15034 ... /home/agent-admin/agent-app/bin/monitor.sh >> /var/log/agent-app/cron.log 2>&1
```

### 의미
- `* * * * *` = **매분**
- 환경변수 명시 주입 (cron이 profile.d를 안 읽으므로)
- `>> ... 2>&1` 표준출력+에러를 cron.log로 누적

### 등록 주체
- agent-admin 본인 셸에서 `crontab -e` → 본인 crontab 에 등록 (sudo 불필요)
- agent-admin은 agent-core 그룹원이라 750 스크립트 실행 권한 있음

### 자주 듣는 질문
- Q. "왜 매분인가? 5분 주기는?"
  - A. "요구사항이 매분이었습니다. 실무에서는 부하 따라 1분/5분/15분 사이 트레이드오프입니다. 매분은 장애 탐지 빠르지만 디스크 부담 ↑."
- Q. "Root crontab이 아니라 agent-admin인 이유?"
  - A. "**최소 권한 원칙**. 일반 계정으로 충분한 작업은 root로 안 합니다. 사고 시 영향 범위가 작아집니다."

## 9. 보너스 — report.sh, log_retention.sh

### report.sh
- monitor.log를 awk로 분석 → CPU/MEM/DISK **평균/최대/최소**
- 시간 옵션: `report.sh "2026-05-13 14:00" "2026-05-13 16:00"`
- awk 한 번으로 끝나서 가볍다.

### log_retention.sh
- 7일 경과 `.log` → gzip 압축
- `/var/log/monitor/agent-app/archive/` 로 이동
- 30일 경과 `.gz` → 삭제
- cron으로 `0 3 * * *` (매일 새벽 3시) 등록 권장

## 11. 평가자가 잘 안 묻지만 알아두면 좋은 것

- **데몬 vs cron 차이**: 데몬은 항상 떠있는 프로세스, cron은 시간 기반 스케줄러. monitor.sh는 매분 짧게 실행되므로 cron이 적합.
- **systemd 대비**: systemd timer로도 같은 일이 가능하지만 cron이 훨씬 단순하고 요구사항이 cron이었음.
- **로그 vs 메트릭**: monitor.log는 **로그**(텍스트 라인). 실무에선 Prometheus/Grafana로 메트릭 보내고 로그는 별도 분리. 본 미션은 학습용 단순 버전.
- **t_secret.key의 t**: test의 t로 추정. 운영용은 별도 분리 가정.
